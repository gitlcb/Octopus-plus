## 一、Octopus 是什么

Octopus 是一个 **LLM API 聚合与负载均衡服务**。简单说，它帮你把"一堆中转站/公益站的一堆 Key"统一成"一个地址 + 一个 Key + 你自己定义的模型名"，再分发给 Claude Code、Codex、Cherry Studio、沉浸式翻译等各种客户端。

本项目在原版 [bestruirui/octopus](https://github.com/bestruirui/octopus) 基础上，额外加入了两块小白最关心的能力：

1. **站点模块**：可以管理你注册的各个中转站，**自动同步可用模型、自动签到**，并把站点的分组/模型物化成渠道。
2. **更完整的协议互转**：可以在 Claude Code 里使用 `OpenAI Response` 甚至 `Gemini` 格式的模型。

> 💡 一句话理解它的设计哲学：**"分组"就是你对外暴露的模型名**。这是和其他工具最不一样、也是小白最容易卡住的地方，请务必看完[第三章](#三必须先懂的-6-个核心概念否则全是坑)。

界面顶部/侧边有 7 个页面，按使用顺序排列：

| 页面 | 作用 |
|------|------|
| **主页** | 花费/请求统计、排行榜、分组健康总览 |
| **站点** | 管理中转站账号，自动同步 + 签到（聚合用户的主战场） |
| **渠道** | 站点渠道（自动投影）+ 普通渠道（手动添加），在这里补全 Key |
| **分组** | **定义对外模型名**，把渠道聚合成一个模型 |
| **价格** | 模型计费价格管理 |
| **日志** | 每次请求的详情、重试、熔断记录 |
| **设置** | 全局配置、API 密钥、账户密码、备份等 |

## 二、5 分钟跑通最短路径

如果你是"导入中转站来聚合使用"的典型场景，整体流程就是下面这张图：

```
1）在中转站里建好你需要的分组 Key（建议「一个分组只建一个 Key」，只建你要用的分组）
        ↓
2）octopus【站点】页 → 新增站点 → 新增账号（填 Access Token）→ 同步
        ↓
3）octopus【渠道】页 → 站点渠道 → 看是否需要「补全 Key」，需要就把中转站对应分组的 Key 复制粘贴进来
        ↓
4）octopus【分组】页 → 新建分组（分组名 = 你将来请求时写的 model 名）→ 添加/自动添加模型
        ↓
5）octopus【设置】→ API 密钥 → 添加密钥（拿到 sk-octopus-... 这串）
        ↓
6）客户端填入：地址 http://你的IP:8080/v1 + 上一步的 Key + 第④步的分组名作为 model
```

> ✅ 完成后这个站点就能正常使用了。后续除非**新增站点**，否则基本不需要再动配置。

如果你的供应商不是中转站、而是直接给了你一个 Base URL + Key（比如智谱 CodingPlan、DeepSeek），那就跳过站点，直接看[第九章 普通渠道](#九普通渠道接入站点之外的供应商)。

## 三、必须先懂的 6 个核心概念（否则全是坑）

| 概念 | 一句话解释 | 小白最容易误解的点 |
|------|-----------|-------------------|
| **站点（Site）** | 一个中转站，记录它的**网址、平台类型、代理** | 网址**只填域名**，不要带 `/v1` 之类路径 |
| **账号（Account）** | 站点下的一份登录凭据，负责**同步 + 签到**；一个站点可挂多个账号 | Access Token 填在**账号**里，不是站点里 |
| **托管渠道 / 投影渠道** | 站点同步后**自动生成**的渠道，会随站点自动维护 | 它**不出现**在"普通渠道"列表里，在"站点渠道"Tab 下看 |
| **普通渠道（Manual）** | 你**手动**添加的渠道（自己有 Base URL + Key 时用） | 与站点渠道是两套，互不影响 |
| **分组（Group）** | **你对外暴露的模型名**——客户端请求的 `model` 必须等于分组名 | **必须先建分组**，否则没有可用模型、也建不了密钥 |
| **密钥（API Key）** | 给客户端用的 `sk-octopus-...`，在「设置 → API 密钥」创建 | 列表里能填的"名称"≠ Key，Key 是创建后生成的那一串 |

> 🔑 **最重要的一句话**：octopus 的"分组"逻辑和别的工具不一样——**分组名就是请求的 model 名**。
> 比如你建一个叫 `gpt-5.5` 的分组，把若干渠道里的 GPT-5.5 模型放进去，客户端就用 `model: gpt-5.5` 来调用。
> 没有分组 = 没有可用模型 = 创建密钥时选不到模型。"获取不到模型 / 建不了 Key"最常见的原因就是还没建分组。

## 四、部署与首次登录

### 4.1 部署（任选一种）

**Docker 直接运行：**
```bash
docker run -d --name octopus -v /path/to/data:/app/data -p 8080:8080 hureru/octopus
```

**docker compose：**
```bash
wget https://raw.githubusercontent.com/Hureru/octopus/refs/heads/dev/docker-compose.yml
docker compose up -d
```

> ⚠️ **镜像一定要用 `hureru/octopus`**（带站点功能的版本）。如果你用的是 `bestrui/octopus` 等原版镜像，是看不到"站点"页面的。Release 二进制、源码编译方式见 [README_zh.md](README_zh.md)。

### 4.2 首次登录

浏览器打开 `http://你的IP:8080`，默认账号：

- 用户名：`admin`
- 密码：`admin`

> ⚠️ **登录提示密码错误？大概率是装过原版、`data` 目录里残留了旧数据**（旧账号密码被保留了下来）。
> 解决：记得旧密码就用旧的；不记得就**清空数据目录后重新部署**。详见 [FAQ Q1](#q1-默认-adminadmin-登录失败)。

### 4.3 立刻改密码

登录后请尽快去 **设置 → 账户设置 → 修改密码**（也可以同时改用户名）。新密码至少 6 位。

> 这里也顺便说明：本项目的登录密钥会持久化到数据库，改密码是安全的。

## 五、第一步：添加站点并同步（站点页）

进入 **站点** 页面。空的时候会有一个"还没有站点 → 新增第一个站点"的引导按钮。新增站点/导入/归档/全量同步/全量签到等入口在**站点页面右上角的工具栏**。

### 5.1 新增站点

点击新增站点，填写：

| 字段 | 说明 |
|------|------|
| **站点名称** | 随便起，方便自己区分，例如「主站 OneAPI」 |
| **平台类型** | 见下表；不确定可留「自动检测」（新建时才有，偶尔检测不准就手动选） |
| **站点地址** | **只填域名**，如 `https://example.com`，**不要带** `/v1`、`/api` 等路径 |
| **手动签到 URL**（可选） | 填了之后可在站点总览里"一键打开"该页面手动签到 |
| **代理** | 直连 / 系统代理 / 代理池（见 [12.x 代理池](#1213-代理池)） |
| **启用站点** | 停用后不再投影托管渠道 |
| **高级设置 → 自定义 Header**（可选） | 给该站点所有请求统一附加的 Header |

支持的**平台类型**：

| 平台 | 说明 |
|------|------|
| **New API** | 最常见的中转站类型（New API / 各类二开） |
| **AnyRouter** | L 站登录的 any 类站点（凭据填 cookie，见下） |
| **One API / One Hub / Done Hub** | 对应同名开源面板 |
| **Sub2API** | sub2api 类站点 |
| **OpenAI / Claude / Gemini** | 官方/直连类 |

### 5.2 新增账号（关键：Access Token 怎么填）

站点建好后会自动弹出"新增账号"。**同步和签到都是账号在做**，凭据填在这里。

**凭据类型**有三种（不同平台可选项不同）：用户名 / 密码、**Access Token**（推荐）、API Key。

根据平台不同，Access Token 的填法是小白最大的坑，重点看这张表：

| 平台 | 推荐凭据 | Access Token 处填什么 | 额外字段 |
|------|---------|----------------------|---------|
| **New API 类** | Access Token | 站点的**「系统访问令牌」**（不是登录密码！） | 必填 **Platform User ID**（站点里的用户 ID） |
| **AnyRouter（any）** | Access Token | **cookie**，格式 `session=MTc1234567890` | — |
| **Sub2API** | Access Token | 站点的 access token | 建议同时填 `refresh_token` 与 `token_expires_at`（F12 拿，401 会自动续期）|
| **OpenAI/Claude/Gemini** | API Key 或 Access Token | 对应的密钥 | — |

> 🔎 **New API 类站点的「系统访问令牌」在哪？**
> 登录中转站 → 个人设置 → 账户管理 → 安全设置 → **系统访问令牌**，生成的那串就是 Access Token。
> **强烈不建议用"账号密码"登录**——很多站点登录后不返回 Access Token，会报 `站点登录成功但未返回 Access Token`。
>
> 🔎 **Platform User ID 是什么？** New API 同步令牌、分组、签到都需要它（就是你在中转站里的用户 ID，例如 `11494`）。用导入功能导入时会尽量自动填好。

账号上还有几个开关：

- **启用账号**：关掉后该账号不参与同步/投影。
- **自动同步**（默认开）：按设置里的周期自动拉取。
- **自动签到**（默认开）：到点自动签到。
- **随机签到**（默认关）：开启后可设「最小签到间隔（小时）」「随机延迟窗口（分钟）」，避免规律化。
- **代理**：可"继承站点代理"，也可单独设。

### 5.3 同步：拉到了什么

点账号上的「同步账号」（或工具栏「全量同步」）后，octopus 会从站点拉取：**分组、Key（脱敏）、模型、余额、今日收入**。

站点卡片上会显示：账号数 / Key 数 / 模型数 / 余额 / 今日收入，以及一个**健康徽章**：

| 徽章 | 含义 |
|------|------|
| 正常 | 一切就绪 |
| 未执行 | 还没同步过，点一下同步 |
| 待配置 | 站点下还没有账号 |
| N 部分同步 | 部分分组同步成功，部分被暂停（见下） |
| N 异常 | 有账号同步或签到失败 |
| N 已停用 / 站点停用 | 有账号/站点被停用 |

**同步状态文案**：`上次同步 时间 · 成功/部分成功/失败 · 具体信息`。常见信息含义：

- **部分成功 · 暂停 N 个缺少可用 Key 的分组投影**：某些分组在你的中转站里**还没有 Key**，octopus 无法投影，于是暂停它们。
  → 解决：去中转站给这些分组建 Key，或在 octopus 渠道页补全 Key（见[第六章](#六第二步在渠道页补全-key站点渠道)），重新同步即可自动恢复。
- **失败 · HTTP 401**：令牌/cookie 失效了，重新获取一个填进去。
- **Cloudflare 保护**：站点触发了 CF。先用浏览器打开站点确认它没挂（服务器挂了也会这样），能正常打开就给账号配代理试试。
- **平台不支持签到**：不是错误，只是该平台（如 sub2api）不支持签到而已，不影响 API 使用。

### 5.4 批量导入（从 All API Hub / Metapi）

站点太多懒得一个个加？用**右上角工具栏的「导入」**：

1. 选择**导入来源**：All API Hub 或 Metapi。
2. **上传 JSON 文件**，或直接**粘贴导出的 JSON**。
3. 点「开始导入」，会按平台 + 站点地址自动创建或复用站点，并显示新增/复用/更新/跳过的数量与告警。

> 注意：Metapi 导入只迁移**站点、账号、Key、分组、模型**，路由策略与下游 Key 会跳过（这正是本项目想摆脱 metapi 复杂路由的初衷）。导入报错见 [FAQ Q10](#q10-导入-all-api-hub--metapi-报错)。

### 5.5 归档与删除

- **归档站点**：从主列表移除、托管渠道下线，但**保留账号/Key/模型**，可在右上角「归档站点」里随时恢复（恢复后默认禁用，启用后会自动重建托管渠道）。
- **删除站点/账号**：连同托管渠道一起删除，不可撤销。

## 六、第二步：在渠道页补全 Key（站点渠道）

进入 **渠道** 页面，顶部有两个 Tab：**站点渠道** 和 **普通渠道**。站点同步出来的渠道都在「站点渠道」里（普通渠道列表默认不显示它们）。

### 6.1 为什么要"补全 Key"

很多中转站**不返回明文 Key**，同步过来的是**脱敏值**（像 `sk-****1234`）。脱敏 Key 没法真正发请求，所以需要你**手动把完整明文 Key 补全**进来。渠道页为此提供了"统一补全面板"。

### 6.2 两种补全方式

**方式 A：顶部「统一补全 Key」按钮**（推荐，集中处理）
- 站点渠道页顶部会出现「统一补全 Key」按钮，带「待补全 N 项」。
- 打开后按 站点 → 账号 → 分组 列出所有待补全的 Key，每行显示：分组、Key 名、当前脱敏值、**输入完整 Key** 的输入框。
- 把对应分组的明文 Key 粘进去，点「保存本账号」，保存后会自动启用并重新参与投影。
- 弹窗里还有「打开令牌管理」可直接跳到该站点的令牌页去复制 Key。

**方式 B：在具体分组里补全 / 创建**
- 在某站点账号面板里，顶部有**分组下拉**和一排按钮。
- 如果某分组**还没建 Key**，会出现「待建 Key N 组」提示，点进去可对支持的站点**快捷创建 Key**（octopus 调用站点接口自动建）。
- 也可对某分组手动「补全/编辑」其投影 Key。

> 💡 建议：**一个站点下每个分组只保留一个 Key，只创建自己需要的分组的 Key**，这样同步和投影最干净。

### 6.3 站点渠道里还能做什么

选中一个分组后，模型表格列出：**模型 / 分组 / 端点格式 / 来源 / Key / 状态 / 最近请求 / 渠道 / 操作**。常用操作：

- **端点格式**（Chat / Response / Embedding）：每个模型可通过"移动至…"切换；自动识别不了的会标「待人工指定」。
- **添加**（自定义模型）：上游 `/models` 没暴露、但其实能用的模型，可以在这里**手动添加**（填模型名 + 端点格式）。
- **投影 / 不投影**：停止/恢复为该分组生成投影渠道。
- **筛选**：仅未正确配置 / 仅有请求历史 / 仅禁用。
- **高级**：对投影渠道做参数覆盖（JSON）。
- **更多 → 重置模型端点格式**：把端点格式恢复为自动识别结果。

**分组状态徽章**：待建（没 Key）/ 待补全（有脱敏 Key 待补全）/ 暂停（系统暂停投影）/ 沿用（最近同步没确认到新模型，沿用上次成功结果）。

> 看到「该分组投影已由系统暂停」别慌——通常是缺少可用 Key 或上游暂时无可用模型，**重新同步成功后会自动恢复**。

## 七、第三步：创建分组 = 对外模型名（分组页）

进入 **分组** 页面，点左上角按钮**创建分组**。这是把渠道聚合成"一个对外模型名"的地方。

### 7.1 表单字段

| 字段 | 说明 |
|------|------|
| **分组名称** | **就是客户端请求时写的 `model` 名**，比如 `gpt-5.5`、`claude-sonnet-4-5` |
| **匹配正则** | 留空则默认按"分组名称模糊匹配"上游模型；也可写正则精确匹配 |
| **首字超时** | 单位秒，仅流式响应生效，0 = 不限制 |
| **会话保持** | 单位秒，会话期间保持用同一渠道，**0 = 禁用** |
| **同通道重试** + **最大重试** | 见下 |
| **负载均衡模式** | 轮询 / 随机 / 故障转移 / 加权分配 |
| **已选模型** | 该分组聚合了哪些"渠道 + 模型" |

**负载均衡模式**：

| 模式 | 说明 |
|------|------|
| 🔄 轮询 | 每次请求依次切换下一个渠道 |
| 🎲 随机 | 每次随机选一个可用渠道 |
| 🛡️ 故障转移 | 优先用高优先级渠道，故障时才切到低优先级 |
| ⚖️ 加权分配 | 按渠道权重比例分配 |

### 7.2 怎么把模型加进分组

在"添加模型"区域：

- **手动添加**：按 渠道 → 模型 勾选（每条会显示来源：站点 / 账号 / 分组，方便区分）。
- **自动添加**（✨ 图标）：把所有"按分组名/正则匹配上"的模型一次性加进来。

> 例：分组名叫 `gpt-5.5`，点"自动添加"，就会把各渠道里名字匹配 `gpt-5.5` 的模型都拉进这个分组。加权模式下还能给每个成员设权重。

### 7.3 自动分组配置（上游新增模型自动归类）

中转站日后可能新增模型。在分组页的**「自动分组配置」**里可以让新模型自动归入匹配的分组：

- **全局默认模式**：开启后，所有站点投影渠道统一用所选匹配方式（关闭/模糊匹配/精确匹配/正则匹配）。开启后，上游同步到新模型时会自动加入匹配的分组。
- **关闭全局**：则上游新增模型时，需要你手动进对应分组点"自动添加"。
- 注意：关闭某渠道的自动分组或全局设置，**不会删除**已存在的分组成员，要清理得手动移除。

> ⚠️ 它**不会**像 metapi 那样"把所有模型一键自动分好组"。你仍然需要**先创建分组**，自动分组只负责把（新）模型归入**已有**分组。

### 7.4 分组健康检查（可选）

在 **设置 → 系统 → 分组健康检查** 开启后，主页会显示分组健康摘要、分组卡片显示健康状态，并可手动运行"标准探活/完整探活"。
（注意：这里是**手动**触发的健康检查，**不是定时自动测活**——多数公益站禁止测活。）

## 八、第四步：创建密钥并接入客户端

### 8.1 创建 octopus 密钥

去 **设置 → API 密钥 → 添加密钥**：

| 字段 | 说明 |
|------|------|
| 名称 | 仅用于区分，**不是 Key 本身** |
| 最大金额 | 该 Key 的费用上限，可设"无限制" |
| 过期日期 | 可设"永不过期" |
| 支持的模型 | **不选 = 无限制**；选了就只能用选中的分组 |
| 是否启用 | — |

> ⚠️ 高频坑：列表里让你填的是**名称**，**真正的 Key 是创建后生成的 `sk-octopus-...` 那一串**，别把名称当成自定义 Key 去用。

创建后用 curl 验证一下（把 IP 和 Key 换成你的）：

```bash
curl http://你的IP:8080/v1/models \
  -H "Authorization: Bearer sk-octopus-你的密钥"
```

能列出模型（也就是你建的**分组名**）就说明通了。

### 8.2 接入常见客户端

**OpenAI SDK / Cherry Studio / 沉浸式翻译**等：
- 地址（Base URL）：`http://你的IP:8080/v1`
- API Key：`sk-octopus-...`
- 模型（model）：填你的**分组名**

```python
from openai import OpenAI
client = OpenAI(
    base_url="http://127.0.0.1:8080/v1",
    api_key="sk-octopus-xxxx",
)
resp = client.chat.completions.create(
    model="gpt-5.5",          # 这里填分组名
    messages=[{"role": "user", "content": "Hello"}],
)
print(resp.choices[0].message.content)
```

**Claude Code**（`~/.claude/settings.json`，注意地址**不带** `/v1`）：
```json
{
  "env": {
    "ANTHROPIC_BASE_URL": "http://127.0.0.1:8080",
    "ANTHROPIC_AUTH_TOKEN": "sk-octopus-xxxx",
    "ANTHROPIC_MODEL": "octopus-sonnet-4-5",
    "ANTHROPIC_SMALL_FAST_MODEL": "octopus-haiku-4-5"
  }
}
```

**Codex**（`~/.codex/config.toml` + `~/.codex/auth.json`）：
```toml
model = "octopus-codex"        # 填分组名
model_provider = "octopus"
[model_providers.octopus]
name = "octopus"
base_url = "http://127.0.0.1:8080/v1"
```
```json
{ "OPENAI_API_KEY": "sk-octopus-xxxx" }
```

> 更多客户端示例见 [README_zh.md 客户端接入](README_zh.md)。

## 九、普通渠道：接入站点之外的供应商

当你**不是用中转站聚合**，而是自己手里有"Base URL + Key"（如智谱 CodingPlan、DeepSeek、某个直连供应商），就用**渠道页 → 普通渠道 → 右上角 + 号**手动添加。

主要字段：

| 字段 | 说明 |
|------|------|
| 渠道名称 | 自定义 |
| 渠道类型 | OpenAI Chat / OpenAI Response / Anthropic / Gemini / 火山引擎 / OpenAI Embedding |
| Base URLs | 只填基础地址，程序按类型自动补 `/chat/completions`、`/responses`、`/messages` 等；可填多个端点做"最低延迟优选" |
| API Key | **可加多个 Key**（实现多 Key 轮询，见 FAQ Q6） |
| 高级设置 | 自定义 Header、渠道代理、参数覆盖（JSON）、自动同步、自动分组、Responses WS 模式、匹配正则、备注 |

加完普通渠道后，**同样要去分组页把它的模型加进某个分组**，才能对外使用。

> 💡 接入要点：
> - 智谱 **CodingPlan** 用普通渠道，渠道类型要选对。
> - DeepSeek（opencode go）若用 Anthropic 格式，可直接加普通渠道；它只吃 chat 格式时，用 Anthropic→Chat 转换也能用。
> - 渠道类型/格式选错是最常见的"配了没反应"的原因。

## 十、协议转换说明

Octopus 支持 **OpenAI Chat / OpenAI Responses / Anthropic** 三种格式互相转换（Gemini 也有转换，但未完全测试）。

**核心规则**：

- **只要把渠道放进分组**，当下游请求的格式与渠道的格式不一致时，**会自动转换**，你不需要做任何额外设置。
- **同模型不要配两种端点格式**。推荐统一设成 **Response**——放进分组后，照样能用 Chat 格式去请求它。
- **透传（passthrough）**：`Response` 格式与 `Anthropic message` 格式在两端一致时会走透传，不被改写；一旦发生格式转换就不再是透传。Codex 的 `fast`、远程压缩端点（Responses compact proxy）在全程 Response 时都会透传。

> 所以"cc 调用 Response 格式的 GPT"基本是没问题的；要用 Gemini 格式的话可以试，遇到 Bug 欢迎反馈。

## 十一、负载均衡 / 熔断 / 重试 / 日志怎么看

### 11.1 重试规则（很多人问）

- 普通渠道里**某个 Key 返回 401**（鉴权失败）→ **不会**再试该渠道的其他 Key，而是**直接切到下一个渠道**。
- **429 / 5xx** 等可重试错误 → 取决于分组的**「同通道重试」**开关：开了就先在**同渠道同 Key**重试（次数 = 最大重试），全部失败后把状态码透传给客户端。

### 11.2 熔断

渠道**连续失败**达到阈值会被熔断一段时间，冷却时间按**指数退避**增长（阈值/基础冷却/最大冷却在 设置 → 熔断器 里调）。

> 怎么知道渠道在不在熔断冷却期？去 **日志** → 点开某条 → **重试详情**，能看到"熔断"标记和**结束熔断的倒计时**。

### 11.3 被动离群退役（可选）

设置 → 被动离群退役：基于真实请求的成败序列，自动**禁用**（不是删除）持续失败的**站点投影渠道**，探活恢复后自动启用。**仅作用于站点投影渠道，默认关闭**。

### 11.4 日志卡片能看什么

每条日志可展开看：错误信息、**首字时间**、总耗时、输入/输出 Token、费用、**重试详情**（每次尝试成功/失败/跳过/熔断）、以及 WS 相关标记（直通/转换/续传/回放/降级等）。

## 十二、设置页各面板一览

| 面板 | 关键项 |
|------|--------|
| **系统** | 代理地址、**统计保存周期（分钟）**、CORS 跨域白名单、Responses WebSocket（含默认模式 直通/转换/关闭）、SSE 心跳、**分组健康检查** 开关 |
| **熔断器** | 熔断触发阈值、基础冷却、最大冷却（指数退避） |
| **被动离群退役** | 仅站点投影渠道、默认关闭，含失败率/最小样本/连续失败/窗口等参数 |
| **渠道同步** | 自动同步间隔（小时）、手动同步 |
| **模型价格** | 自动更新间隔（小时）、手动更新（数据来自 models.dev） |
| **站点自动化** | 站点自动同步间隔（小时）、自动签到间隔（小时）、手动全量同步 / 全量签到 |
| **日志设置** | 启用历史日志、日志保存天数、清空历史日志 |
| **备份 / 恢复** | 导出（可选含日志/统计，含日志强制 ZIP）、导入（增量） |
| **API 密钥** | 创建/管理 `sk-octopus-*`（见[第八章](#八第四步创建密钥并接入客户端)） |
| **账户设置** | 修改用户名 / 密码 |
| **版本信息** | 当前/最新版本、一键更新（更新前请先备份） |
| **外观** | 主题（浅色/深色/跟随系统）、语言 |

### 12.13 代理池

设置里可打开**代理池**集中管理可复用的代理配置（支持 `http / https / socks / socks5`）。站点、站点账号、普通渠道都能从代理池里选同一个代理，省得到处重复填。代理模式有：继承 / 直连 / 系统代理 / 代理池。旧版本的代理会自动迁移到代理池。

## 十三、高频问题 FAQ

> 汇总最常见的使用问题。

### Q1. 默认 admin/admin 登录失败？
大概率是**装过原版、`data` 目录残留了旧数据**，旧账号密码被保留了。记得旧密码就用旧的；不记得就**清空数据目录后重新部署**。干净环境下 admin/admin 一定能登录。

### Q2. 已经有模型了，为什么创建 Key 时说没有可用模型？
因为**还没建分组**。octopus 里"分组名 = 可用的模型名"，**先去分组页新建一个分组**，创建 Key 时就能选到了。

### Q3. 站点同步成功了，渠道里却"没有新建/没有可用 Key"？
- 若**分组都获取不到** → 多半是站点同步报错（看同步状态信息）。
- 若**有分组但 Key 是脱敏的** → 去渠道页用**「统一补全 Key」**把明文 Key 补上。
- 中转站里那个分组**压根没建 Key** → 先去站点建 Key，或在 octopus 里"快捷创建 Key"。

### Q4. Access Token 到底填什么？
- **New API 类**：填站点「个人设置 → 账户管理 → 安全设置 → **系统访问令牌**」生成的那串，**不要用账号密码登录**；并且 New API 还要填 **Platform User ID**。
- **any（AnyRouter）**：填 **cookie**，格式 `session=MTc1234567890`，平台类型记得选 **AnyRouter** 而不是 New API。
- 除 any 外的站点都填系统访问令牌，**不要填 cookie**。

### Q5. 站点网址要怎么填？
**只填域名**，例如 `https://wzw.pp.ua`，**不要带** `/v1` 等路径。

### Q6. 一个渠道有多个 Key，想轮流用怎么配？
用**普通渠道**，在里面加多个 Key；然后把该渠道所在**分组的「会话保持」设为 0**。这样各 Key 的累计调用金额会趋于一致，如果每次成本差不多，就接近轮流使用的效果。会话保持不为 0 会"黏"住一个 Key。普通渠道面板里能看到每个 Key 的使用金额，多请求几次就能看出来。

### Q7. 401 / 429 会自动切换 Key 吗？
普通渠道某个 Key 返回 **401 不会**再试其他 Key，直接切下一个渠道；**429** 看分组有没有开「同通道重试」，开了就同渠道同 Key 重试。

### Q8. 日志里的模型名和我配的不一样、而且计费为 0？
不是 bug。日志里的模型名按**上游响应里的 `model` 字段**显示。如果上游返回的是带日期后缀的变体名（如 `gpt-5.4-mini-2026-03-17`）且你没给它定价，就会计费为 0——去**价格**页给这个名字**手动定价**即可。这种上游不多，可以接受。

### Q9. 同步报错 `decode response failed` / HTTP 401 / Cloudflare？
- **401**：令牌/cookie 失效，重新获取。
- **decode response failed**：多半是站点地址带了多余路径或类型选错，确认只填域名、类型选对。
- **Cloudflare 保护**：先用浏览器（Ctrl+F5）确认站点没挂，服务器挂了也会报 CF；站点正常的话给账号配代理再试。
- **平台不支持签到**：不是异常，sub2api 等不支持签到的站点同步只是 401/跳过签到，不影响使用。

### Q10. 导入 All API Hub / Metapi 报错？
- 确认导出来源选对（All API Hub ↔ Metapi 别搞反）、JSON 完整。
- `cannot unmarshal string into ... version of type int`：来自旧版 metapi 的版本号字段类型问题，升级 metapi 到较新版本再导出。
- 导入后若仍有异常站点，可（不带日志）导出数据后重新部署一个干净实例再导入。

### Q11. 自动分组配置有什么用？是不是建了它就不用手动建分组了？
不是。你仍要**先创建分组**。自动分组只负责在上游**新增模型**时，把新模型按"分组名/正则"自动归入**已有**分组（开启全局默认模式即可）；关闭则需手动进分组点"自动添加"。它不会像 metapi 那样一键把所有模型分好组。

### Q12. 同一个模型，OpenAI Chat 和 Response 想都能用怎么配？
不需要配两个。**统一设成 Response**，放进分组后用 Chat 格式照样能请求（会自动转换）。没用的 Key 请到**上游站点**删除。

### Q13. 拉取模型/对话不走代理？
拉模型和 API 请求都会走 **站点/账号配置的代理**。如果发现没走，检查账号代理是不是设成了"继承站点代理"而站点又没配代理；必要时直接在账号上选代理池里的代理。

### Q14. 打开分组/渠道页有点卡？
通常是首次加载渠道里的模型列表/分组时的瞬时卡顿，缓存后会好转；日志数据量很大时也会更重一些，可在设置里控制日志保留天数、定期清理。

### Q15. 视觉/图片模型分发后"看不到图"？
先确认**上游供应商本身**配置正常（同样的图在原 API 能读）。也出现过"看不到图"其实是模型自己"以为看不到"、多试几次就正常的情况——优先排查供应商与端点格式配置。

## 十四、性能与运维

- **资源占用**：内存约 50–60 MB；长期使用数据库文件大致在 GB 级（连续几个月约 3GB 左右）。
- **统计写入**：统计先存在内存，按「统计保存周期」批量写库。
  > ⚠️ **退出务必用正常方式**（`Ctrl+C` 或 `SIGTERM` / `docker stop`），**不要 `kill -9`**，否则内存里的统计会丢失。
- **升级**：可放心升级；**日志数据量很大（几千条以上）时，升级会做数据库迁移，可能耗时几分钟**，属正常现象。更新前建议先在「设置 → 备份」导出。
- **后台任务**：统计持久化、模型同步、价格更新、站点同步/签到等都在后台定时执行（周期可在设置里调）。
